# 贪吃蛇游戏 — 项目规范

## 开发纪律

### 测试先行（TDD）
- **每个功能模块实现后，必须立即编写并运行测试**，严禁"全部写完再统一测试"
- 流程：写测试 → 确认失败 → 写实现 → 确认通过 → 提交
- 运行测试一律使用 `bash scripts/test.sh`，禁止直接调用 `pytest`

### 代码可读性
- **关键逻辑必须有中文注释**，包括：模块用途说明、类/方法的 docstring、复杂算法的行内注释
- 函数和方法必须标注参数类型和返回值类型（type hints）
- 变量命名要自解释，禁止 `a`、`b`、`tmp`、`data` 等无意义名称
- 每个代码文件不超过 **300 行**，超过必须拆分

### 代码质量
- **禁止死代码**：删除不再使用的函数、变量和导入，不留注释掉的旧代码
- **禁止重复代码**：相同逻辑出现 2 次以上，必须抽取为公共函数或类
- **禁止 magic number**：所有数值常量必须定义在 `src/core/state.py` 中，禁止在业务代码里硬编码数字
- **单向依赖**：实体层不依赖系统层，系统层不依赖渲染层，渲染层不修改游戏状态

### 错误处理
- 对外部资源（文件读写、JSON 解析）必须使用 `try/except` 并给出合理的降级行为
- 禁止使用空 `except:`，必须指定具体异常类型
- 错误信息使用中文

### Git 规范
- 每个 Task 完成后必须提交，提交信息使用中文
- 格式：`feat:` / `fix:` / `refactor:` / `docs:` / `test:` + 中文描述
- 禁止一次性提交大量不相关的改动

### 运行与调试
- 运行游戏：`bash scripts/run.sh`
- 运行测试：`bash scripts/test.sh`
- 依赖管理只使用 `uv`，禁止 `pip`、`poetry`、`conda`
- 所有日志输出到 `logs/` 目录

### 文档
- 新增或修改公共接口时，同步更新对应的 docstring
- 架构决策记录在 `docs/` 目录下
